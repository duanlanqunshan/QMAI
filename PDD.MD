# PDD.MD

# Novel Wiki Writer 产品设计文档

版本：v0.1  
定位：基于 llm_wiki UI 与知识系统改造的长篇小说 AI 写作桌面端软件  
技术方向：Rust 桌面端，尽量复用 llm_wiki 原有能力，最小代码改动

---

## 1. 产品愿景

Novel Wiki Writer 的目标是把 llm_wiki 的知识库系统改造成一个面向小说的长篇写作系统。

它要解决的问题不是“让 AI 随便写一章”，而是：

> 让 AI 在写每一章之前都能读取正确上下文，在写完每一章之后都能把新内容沉淀为可检索、可图谱化、可审阅、可复用的小说记忆。

最终目标是支持 200 万～300 万字长篇小说创作。

---

## 2. 背景与问题

普通 AI 小说写作工具存在以下问题：

1. 长篇上下文无法完整输入。
2. AI 写到后期容易忘记前文。
3. 人物性格容易变化。
4. 时间线容易混乱。
5. 伏笔容易丢失。
6. 角色知道了不该知道的信息。
7. 新增设定没有登记，后续无法追踪。
8. 用户无法查看人物、事件、伏笔之间的关系。
9. AI 生成的错误内容容易污染后续创作。

本产品通过以下方式解决：

- 大纲作为最高规则
- 每次写作前自动提取上下文
- 每章写完后自动提取章节记忆
- 使用章节快照代替全文硬塞进上下文
- 使用小说图谱管理人物、事件、伏笔、地点、物品关系
- 使用审阅与检查系统防止崩坏
- 使用向量搜索、关键词搜索、图谱搜索混合检索
- 区分草稿与正式章节，避免错误内容污染 Canon 正史库

## 3. 产品目标

### 3.1 核心目标

- 基于 llm_wiki 最小改动完成小说化改造。
- 保留原 UI 基本结构。
- 将知识库能力转化为小说章节与记忆能力。
- 每次写作前自动生成上下文包。
- 每章保存后自动提取关键内容。
- 自动生成并更新小说图谱。
- 支持长篇小说的持续写作。
- 降低 AI 幻觉和长篇崩坏概率。

### 3.2 非目标

第一阶段不做：

- 不重写整个 llm_wiki
- 不重做 UI
- 不做云端多人协作
- 不做复杂出版排版
- 不做全自动无人工确认写完整本书
- 不承诺完全消除 AI 幻觉
- 不承诺小说一定好看

## 5. 产品原则

### 5.1 最小改动原则

本产品基于 llm_wiki 改造，原则是：

```text
能复用，不重写。
能配置，不改代码。
能适配，不侵入。
能新增模块，不重构原模块。
```

### 5.2 人工确认原则

AI 生成内容默认不可信。

必须由用户确认后，才能：

- 保存为正式章节
- 更新 Canon 正史库
- 更新人物状态
- 更新伏笔状态
- 更新图谱
- 更新搜索索引
- 更新向量索引

### 5.3 结构化记忆原则

长篇小说不能只保存正文。

每章都必须沉淀为：

- 章节摘要
- 人物状态
- 事件
- 时间线
- 伏笔
- 角色认知
- 图谱关系
- Canon 规则

### 5.4 混合检索原则

不得只依赖 embedding。

必须综合：

```text
最近章节窗口
+ 大纲
+ Canon 规则
+ 关键词搜索
+ 向量搜索
+ 图谱搜索
+ 章节快照
```

## 6. 信息架构

### 6.1 UI 区域

用户指定 UI 区域如下：

```text
1. 左侧主导航
2. 项目内容列表
3. AI 会话列表
4. AI 聊天区域
5. 小说写作生成 / 小说档案详情区域
```

### 6.2 区域 1：左侧主导航

功能：

- 章节库
- 大纲库
- 搜索
- 图谱
- 审稿
- 检查
- 设置

改造要求：

- 位置不变
- 图标尽量不变
- 文案小说化
- 不重做交互

### 6.3 区域 2：项目内容列表

展示：

- 章节
- 大纲
- 人物
- 地点
- 组织
- 物品
- 事件
- 伏笔
- 概念
- 时间线

复用 llm_wiki 原 Entity / Page / Source 列表。

### 6.4 区域 3：AI 会话列表

展示 AI 对话历史。

会话类型：

- 大纲生成
- 章节生成
- 章节续写
- 章节改写
- AI 审稿
- 设定补全
- 伏笔整理
- 图谱分析

### 6.5 区域 4：AI 聊天区域

用户输入自然语言需求。

示例：

```text
继续写第 30 章。

找出还没回收的伏笔。
帮我重写这一段，让节奏更紧。
```

系统需要识别任务类型，然后调用不同能力。

### 6.6 区域 5：小说写作生成 / 详情区域

根据当前选择显示：

- 章节正文
- 生成草稿
- 章节快照
- 人物档案
- 伏笔档案
- 时间线事件
- 图谱详情
- 审稿结果
- 检查结果

这是小说版核心展示区域。

## 7. 功能需求

### FR-001 小说模式开关

系统应支持小说模式。

小说模式开启后：

- UI 文案小说化
- 知识库语义变为章节库
- 文件语义变为大纲库
- Review 变为 AI 审稿
- Graph 变为小说图谱
- Search 变为剧情搜索
- Lint 变为连贯性检查

小说模式关闭后：

- 原 llm_wiki 行为尽量保持不变

### FR-002 大纲管理

系统应支持：

- 导入大纲
- 编辑大纲
- AI 生成大纲
- 总大纲
- 分卷大纲
- 章节细纲
- 人物设定
- 世界观设定
- 伏笔计划
- 时间线设定

### FR-003 章节生成

系统应支持：

- 生成章节草稿
- 续写章节
- 扩写章节
- 改写章节
- 润色章节
- 根据审稿意见修改章节

### FR-004 上下文包生成

每次调用 LLM 写作前，系统必须生成上下文包。

上下文包至少包含：

- 当前任务
- 当前章节目标
- 必须遵守的大纲
- 最近剧情摘要
- 上一章结尾
- 当前人物状态
- 角色认知状态
- 当前伏笔状态
- 时间线
- 相关地点 / 组织 / 物品
- 禁止违背规则
- 写作风格

### FR-005 草稿管理

AI 生成内容默认为草稿。

草稿支持：

- 预览
- 编辑
- 重新生成
- 审稿
- 废弃
- 保存为正式章节

草稿不得更新正式记忆库。

### FR-006 正式章节保存

用户确认后，草稿可以保存为正式章节。

保存后系统必须：

- 保存正文
- 生成章节快照
- 提取实体
- 提取事件
- 提取状态变化
- 提取伏笔变化
- 更新时间线
- 更新图谱
- 更新搜索索引
- 更新向量索引

### FR-007 章节摄取

章节摄取需要提取：

```text
章节摘要
出场人物
出场地点
出场组织
出场物品
关键事件
人物状态变化
人物关系变化
角色认知变化
新增伏笔
推进伏笔
回收伏笔
新增设定
时间线事件
冲突变化
章节结尾钩子
图谱节点
图谱关系边
```

### FR-008 小说图谱

系统应支持小说图谱展示。

节点类型：

- 人物
- 地点
- 组织
- 物品
- 事件
- 章节
- 大纲
- 伏笔
- 秘密
- 冲突
- 时间点
- 正史规则

关系类型：

- 出场于
- 发生于
- 持有
- 敌对
- 合作
- 怀疑
- 隐瞒
- 知道
- 不知道
- 推进伏笔
- 回收伏笔
- 新增伏笔
- 导致
- 揭示
- 影响

### FR-009 AI 审稿

系统应支持对章节草稿或正式章节审稿。

审稿维度：

- 大纲一致性
- 人设一致性
- 时间线一致性
- 地点一致性
- 伏笔一致性
- 角色认知一致性
- 能力体系一致性
- 节奏
- 情绪
- 爽点
- 章节钩子

### FR-010 连贯性检查

系统应输出结构化检查结果：

```json
{
  "id": "lint-001",
  "severity": "error",
  "type": "timeline_error",
  "message": "本章时间线与上一章冲突",
  "evidence": "正文证据",
  "related_memory": "相关章节快照",
  "suggestion": "修改建议"
}
```

### FR-011 剧情搜索

系统应支持：

- 关键词搜索
- 语义搜索
- 图谱搜索
- 混合搜索

搜索范围：

- 大纲
- 章节
- 章节快照
- 人物档案
- 地点档案
- 组织档案
- 物品档案
- 伏笔档案
- 时间线
- 审稿记录

### FR-012 设置

设置应包含：

- 聊天模型
- 写作模型
- 审稿模型
- 摘要模型
- 提取模型
- 嵌入模型
- 上下文 token 预算
- 最近章节数量
- 检索 Top K
- 是否启用向量检索
- 是否启用关键词检索
- 是否启用图谱检索
- 是否启用自动章节摄取
- 是否启用保存前审稿

## 8. 核心流程设计

### 8.1 新建小说项目流程

```text
用户新建项目
↓
输入小说名、题材、目标字数
↓
选择是否导入大纲
↓
如果无大纲，调用 AI 生成大纲
↓
系统摄取大纲
↓
提取初始人物、地点、势力、伏笔、时间线
↓
生成初始小说图谱
```

### 8.2 写作流程

```text
用户输入写作需求
↓
主脑识别任务类型
↓
上下文引擎提取记忆
↓
生成上下文包
↓
LLM 生成章节草稿
↓
展示到区域 5
↓
用户编辑 / 审稿 / 重新生成
↓
用户确认
↓
保存为正式章节
```

### 8.3 章节保存流程

```text
草稿
↓
用户确认
↓
正式章节
↓
章节摄取
↓
章节快照
↓
实体与事件提取
↓
人物状态更新
↓
伏笔状态更新
↓
时间线更新
↓
图谱更新
↓
索引更新
```

### 8.4 下一章上下文流程

```text
用户请求写下一章
↓
读取当前章节细纲
↓
读取上一章快照
↓
读取最近 N 章摘要
↓
检索相关人物
↓
检索相关伏笔
↓
检索相关地点
↓
检索相关时间线
↓
图谱扩展相关节点
↓
生成上下文包
↓
调用 LLM
```

## 10. 上下文引擎设计

### 10.1 输入

```json
{
  "user_request": "继续写第128章",
  "current_chapter": 128,
  "task_type": "write_chapter",
  "novel_id": "novel-001"
}
```

### 10.2 检索步骤

```text
1. 读取总大纲
2. 读取当前卷大纲
3. 读取当前章节细纲
4. 读取上一章正文或快照
5. 读取最近 N 章摘要
6. 根据任务抽取关键词
7. 关键词搜索相关记忆
8. 向量搜索相关记忆
9. 图谱搜索相关节点
10. 读取 Canon 规则
11. 读取人物状态
12. 读取伏笔状态
13. 读取角色认知状态
14. 合并、去重、排序
15. 控制 token 数量
16. 生成上下文包
```

### 10.3 排序规则

优先级建议：

```text
1. 用户明确指定内容
2. 当前章节细纲
3. 上一章结尾
4. Canon 正史规则
5. 当前出场人物状态
6. 当前伏笔状态
7. 最近章节摘要
8. 图谱一跳关系
9. 向量高相关内容
10. 关键词搜索内容
11. 远期历史摘要
```

## 11. 章节摄取设计

```json
{
  "chapter_id": "chapter-128",

  "previous_memory": "上一章快照与相关记忆"
```

### 11.2 提取任务

LLM 或提取器需要输出结构化 JSON：

```json
{
  "summary": "",
  "characters": [],
  "locations": [],
  "organizations": [],
  "items": [],
  "events": [],

  "relationship_changes": [],
```

### 11.3 校验

摄取结果入库前必须校验：

- JSON 是否合法
- 必填字段是否存在

- 是否引用了不存在的实体
- 是否包含明显与 Canon 冲突的内容
- 是否需要人工确认

## 12. AI Prompt 模板

### 12.1 章节生成 Prompt

```text
你是小说写作助手。你必须严格遵守【小说上下文包】。
不要编造与上下文冲突的设定。
不要让角色知道他不该知道的信息。
不要提前回收未计划回收的伏笔。
不要改变人物核心性格。
请根据当前章节目标生成正文。

输出要求：
- 只输出章节正文
- 不要解释
- 保持指定文风
- 结尾保留钩子
```

### 12.2 章节摄取 Prompt

```text
你是小说章节记忆提取器。
请从章节正文中提取结构化记忆。
只提取正文明确发生或可合理确定的内容。
不要把推测当成事实。
不要把草稿错误写入 Canon。
请输出合法 JSON。
```

### 12.3 审稿 Prompt

```text
你是长篇小说审稿助手。
请根据大纲、章节快照、人物状态、伏笔状态、时间线和角色认知状态检查本章。
重点检查：
1. 人设是否崩坏
2. 时间线是否冲突
3. 伏笔是否错乱
4. 角色是否知道了不该知道的信息
5. 是否违背大纲
6. 是否新增未登记设定
7. 章节节奏是否拖沓
8. 结尾是否有钩子

请输出结构化审稿意见。
```

## 13. 非功能需求

### NFR-001 性能

系统应能处理：

- 1000+ 章节
- 200 万～300 万字正文
- 大量章节快照
- 大量人物与伏笔节点
- 大量图谱关系

### NFR-002 稳定性

- 模型调用失败不能导致正文丢失
- 摄取失败不能破坏已有记忆库
- 图谱更新失败应可重试
- 索引更新失败应可重建

### NFR-003 可维护性

- 小说改造代码尽量集中在 novel 模块
- 不要散落大量硬编码判断
- 关键流程有测试
- Prompt 模板集中管理

### NFR-004 隐私

- 不在代码中硬编码 API Key
- 不上传用户小说内容到未配置的服务
- 用户应能配置模型供应商
- 用户应能关闭自动发送内容

### NFR-005 可迁移

- 项目数据尽量可导出
- 章节正文可保存为 Markdown
- 章节快照可保存为 JSON / Markdown
- 图谱数据可重建

## 16. 风险与对策

### 16.1 风险：改动过大导致原项目不可维护

对策：

- 严格遵守最小改动原则
- 小说功能集中在 novel 模块
- 使用适配层
- 保留原功能

### 16.2 风险：AI 提取错误污染记忆库

对策：

- 草稿不入库
- 用户确认后入库
- 提取结果可审阅
- 摄取前做校验
- 支持回滚和重建索引

### 16.3 风险：Embedding 检索漏掉关键内容

对策：

- 使用混合检索
- 结合关键词、图谱、最近章节、Canon 规则
- 关键设定强制加入上下文包

### 16.4 风险：上下文包过长

对策：

- 设置 token 预算
- 按重要性排序
- 摘要压缩
- 最近章节优先
- Canon 规则优先

### 16.5 风险：小说质量不可控

对策：

- 审稿
- 节奏检查
- 人工确认
- 支持重写和局部修改
- 提供章节目标与风格控制

## 17. 验收标准

MVP 验收需要满足：

1. 软件 UI 基本保持原 llm_wiki 布局。
2. 小说模式下文案正确映射。
3. 可以导入或生成大纲。
4. 可以基于大纲生成章节草稿。
5. 生成章节前能自动生成上下文包。
6. 用户确认后可以保存为正式章节。
7. 正式章节保存后能生成章节快照。
8. 章节快照包含摘要、人物、事件、伏笔、时间线等信息。
9. 图谱能展示章节相关人物 / 事件 / 伏笔关系。
10. 搜索能搜到章节关键内容。
11. 审稿能发现基础人设、时间线、伏笔问题。
12. 草稿不会污染正式记忆库。
13. 原 llm_wiki 主要功能未被破坏。
14. 代码改动符合最小改动原则。
15. 项目可以运行、检查和测试。

## 18. 一句话总结

Novel Wiki Writer 的本质不是普通 AI 写作聊天工具，而是：

> 基于 llm_wiki 的小说记忆型 AI 写作系统。  
> 它通过“写前上下文提取 + 写后章节摄取 + 小说图谱 + 审稿检查 + 人工确认”，让 AI 能够长期、稳定地辅助创作 200 万～300 万字长篇小说。
