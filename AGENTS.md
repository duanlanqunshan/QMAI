# 编码与本地开发执行准则

## 1. 先思考，再写代码

**不要想当然。不要掩盖困惑。把假设和权衡摆到台面上。**

在开始实现之前：

* 明确写出你的假设。若不确定，就提问。
* 如果存在多种理解方式，列出来，不要悄悄自行选择一种。
* 如果有更简单的方案，要主动说明。必要时要提出不同意见，而不是盲从。
* 如果有地方不清楚，就暂停。明确指出困惑点，并提出问题。

## 2. 简单优先

**只写解决问题所需的最少代码。不要加入任何臆测性的内容。

* 不要添加用户没要求的功能。
* 不要为一次性代码做抽象。
* 不要加入用户未要求的“灵活性”或“可配置性”。
* 不要为实际上不会发生的场景编写错误处理。
* 如果你写了 200 行，但其实 50 行就能解决，那就重写。

随时问自己一句：

**“一个资深工程师会不会觉得这写得太复杂了？”**

如果答案是会，那就继续简化。

## 3. 外科手术式修改

**只改必须改的地方。只清理你自己造成的问题。**

在修改现有代码时：

* 不要顺手“改善”旁边无关的代码、注释或格式。
* 不要重构那些没有坏掉的部分。
* 保持与现有风格一致，即使你自己会写得不一样。
* 如果你发现了无关的死代码，可以提出来，但不要擅自删除。

当你的改动产生了遗留物时：

* 删除那些因为你的改动而变成未使用的 import、变量、函数。
* 不要删除原本就存在的死代码，除非用户明确要求。

判断标准是：

**每一行改动，都应当能直接追溯到用户的请求。**

## 4. 以目标为导向执行

**先定义成功标准，再循环推进，直到验证通过。**

把任务转化成可验证的目标：

* “加输入校验” → “先为非法输入写测试，再让测试通过”
* “修复这个 bug” → “先写一个能复现问题的测试，再让它通过”
* “重构 X” → “确保改动前后测试都通过”

对于多步骤任务，先给出一个简短计划：

1. [步骤] → 验证方式：[检查项]
2. [步骤] → 验证方式：[检查项]
3. [步骤] → 验证方式：[检查项]

强成功标准能让你独立推进并持续验证。弱成功标准（例如“把它弄好”）则会迫使你不断回来确认。



# 本地开发与环境操作硬性规则

以下规则属于高优先级执行约束。当它们与“快速完成任务”发生冲突时，优先遵守这些安全与操作规则。

## 5. 严禁未经用户同意删除电脑文件

* 包括删除、覆盖、批量移动、清空、卸载、重置等可能造成数据丢失的操作。

## 6. 面向用户的提示语、弹窗、确认信息、风险说明、操作说明，必须全部使用中文。

## 7. 修改本地设置、执行命令、调整软件环境前，必须先评估影响

* 要先判断是否会影响本机其他软件、其他项目、全局环境或版本冲突；优先选择局部、低风险方案。

## 8. 完成任务后，尽量反馈：

*是否完成*是否做了测试验证*有哪些风险*如果有代码改动，是否已 git 提交*下一步建议是什么

任务完成之后给出是否有继续优化的内容，如果有给出建议，如果没有则给出相应提示。另外判断当前结束后是否需要回答优化内容，这个回答不是必须要做的。比如需求澄清阶段这个就没有必要去做，或者写计划阶段也是没有必要的。只有当一个完整的任务完成后才判断是否要给出继续优化的内容。

## 9.添加功能

增加功能时，总是会出现这个功能增加上之后其他的功能可能就会丢失，所以必须要预防这个问题。
必须要使用以下逻辑！

   1. 先把“能正常工作的版本”固定下来

2. 新功能一定放到分支里做
3. 新功能完成后，不要马上打包，先跑旧功能
4. 新增功能或要修改时，要修改现有功能的逻辑。
5. 不要重构无关代码。
6. 不要删除已有函数。
7. 修改前先说明你准备改哪些文件。
8. 新增功能完成后，列出所有改动点。
9. 用 Git 做版本管理 + 用测试/打包清单防止功能回退！
10. main 分支：永远保持稳定可打包
11. feature 分支：每次只加一个新功能
12. 改完先跑源码
13. 再跑旧功能测试
14. 确认没问题再打包 exe
15. 打包成功后再合并回 main
