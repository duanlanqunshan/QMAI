# 章节长按拖动重写设计文档

> 日期：2026-05-28
> 分支：待创建功能分支
> 方案：点击选中 + 长按拖动重排

---

## 1. 背景与目标

当前章节移动已经反复修改多次，但旧逻辑仍然存在几个核心问题：

1. 长按进入移动的时机不稳定。
2. 拖动目标识别不稳定，容易出现“看起来拖了，结果没按预期落位”。
3. 长按过程中容易和文本选中、复制、右键等交互打架。

这次不继续在旧逻辑上补丁，而是直接重做一套更清晰的章节拖动方案。

本次目标：

1. 保留“点击章节即可选中和打开”的主行为。
2. 只有已选中的章节，长按后才能进入拖动。
3. 拖动时提供明确的插入位置反馈，松手后按最终位置重排。
4. 只修改 `chapter_number`，不改标题、不改文件名、不改路径。
5. 保持右键菜单、重命名、删除等现有功能继续可用。

---

## 2. 交互设计

### 2.1 正常状态

章节列表默认保持现有主行为：

- 单击章节：选中并打开章节。
- 右键章节：打开右键菜单。
- 删除、重命名：保持现有逻辑。
- 未选中章节长按：不进入移动。

### 2.2 进入拖动

只有当前已选中的章节，才允许通过长按进入拖动状态。

进入条件：

1. 当前列表模式为章节列表。
2. 当前行已经是 `selectedFile`。
3. 左键按下后持续约 300ms。
4. 期间没有提前结束按压。

这样可以把“普通点击打开章节”和“长按开始移动”明确分开。

### 2.3 拖动反馈

进入拖动后，界面提供最小但清晰的反馈：

- 被拖动的章节行进入拖动态。
- 列表中显示一条插入位置线，表示松手后会插入到哪里。
- 当前悬停目标行可以有轻量高亮。
- 整个章节列表继续禁止文本选中和系统复制触发。

这次不做拖动手柄，不增加额外按钮，不做隐藏“移动模式”。

### 2.4 松手结果

拖动结束时按以下规则处理：

1. 如果最终位置没有变化，不写文件。
2. 如果最终位置发生变化，按新的列表顺序重排章节编号。
3. 重排完成后退出拖动态并刷新列表。

---

## 3. 数据规则

### 3.1 顺序来源

章节显示顺序继续以 `chapter_number` 为主。

排序优先级：

1. 优先使用 frontmatter 中的 `chapter_number`。
2. 若缺失，再回退到现有标题中的章节号提取逻辑。
3. 再无法提取时，回退到标题字典序。

### 3.2 写回策略

这次不再做“交换两个章节编号”的旧思路，而是按拖动后的最终顺序，统一重写受影响章节的 `chapter_number`。

只修改：

- `chapter_number`

不修改：

- `title`
- 文件名
- 文件路径
- 正文内容

这样可以保证：

1. 用户看到的章节标题不会变。
2. 目录结构和引用路径不被扰动。
3. 列表顺序只由章节编号驱动。

### 3.3 重排语义

本次采用“插入到目标位置”的语义：

- 拖动源章节从原位置取出。
- 根据插入线位置插入到新位置。
- 对新顺序中的相关章节重新编号。

这比旧的“悬停一行就交换”更符合真实拖动排序的预期。

## 4. 实现边界

### 4.1 主要修改文件

核心修改限定在：

- `src/components/layout/knowledge-tree.tsx`
- `src/components/layout/knowledge-tree.long-press.test.tsx`

必要时补少量文案到：

- `src/i18n/zh.json`
- `src/i18n/en.json`

### 4.2 明确删除的旧逻辑

以下旧移动链路需要整体移除或重写：

- 旧的“长按后直接按 hover 行交换编号”逻辑
- 基于 `dragSource` / `dragHover` 的交换型移动语义
- 依赖 `elementFromPoint` 命中某一行后立即确定交换目标的旧策略

不是删除“长按”这个入口，而是删除“长按后交换位置”的旧实现。

### 4.3 新状态

建议保留简洁状态模型：

- `dragSourcePath: string | null`
- `dragInsertIndex: number | null`
- `isDragging: boolean`
- 长按计时器与当前 pointer 信息

### 4.4 新增行为函数

建议把重排写回独立成清晰函数，例如：

- `executeChapterReorder(sourcePath, targetIndex)`

职责：

1. 找到当前章节列表顺序。
2. 计算拖动后的新顺序。
3. 只更新受影响章节的 `chapter_number`。
4. 顺序写回原文件。
5. 失败时尽量回滚已写入内容。
6. 刷新页面列表与文件树。

## 5. 兼容性与风险控制

### 5.1 必须保持可用的现有能力

以下功能不能因为重做拖动而回退：

- 点击打开章节
- 已选中高亮
- 桌面右键菜单
- 重命名
- 删除
- 文件树刷新

### 5.2 主要风险

1. 点击与长按的边界处理不好，会误伤正常打开章节。
2. 拖动插入位置计算不准，会导致落位错误。
3. 统一重排 `chapter_number` 时，可能写坏多章顺序。
4. 右键菜单和长按拖动可能互相干扰。

### 5.3 控制策略

1. 用测试固定“未选中章节不能开始拖动”。
2. 用测试固定“已选中章节长按后才进入拖动”。
3. 用测试固定“松手后按新顺序重写编号，不改标题和路径”。
4. 用测试固定“右键菜单仍可用”。

## 6. 测试与验收

### 6.1 先写失败测试

按 TDD 执行，先把下面几类行为固定成失败测试：

1. 点击章节仍然正常打开。
2. 未选中章节长按不会触发移动。
3. 已选中章节长按后进入拖动。
4. 拖动到新位置后，按新顺序重写 `chapter_number`。
5. 标题和文件路径保持不变。
6. 如果位置没变，不写文件。
7. 桌面右键菜单继续可用。

### 6.2 代码验证

至少执行：

```powershell
npx vitest run src/components/layout/knowledge-tree.long-press.test.tsx
npm run test:mocks
npm run typecheck
```

### 6.3 页面验证

本地页面验证目标：

1. 点击章节可正常打开。
2. 选中章节后长按，可拖动到新位置。
3. 拖动时有明显落位反馈。
4. 松手后顺序变化正确。
5. 右键菜单、重命名、删除没有明显回退。

---

## 7. 不做的事情

本次明确不做：

- 不重构整个 `KnowledgeTree`
- 不改章节命名规则
- 不改章节重命名流程
- 不扩展到大纲移动
- 不增加新的“移动模式”按钮
- 不做打包前的无关 UI 优化

---

## 8. 推荐实施顺序

1. 先把旧测试改成新的长按拖动失败测试。
2. 跑测试，确认按新预期失败。
3. 重写章节长按拖动链路。
4. 跑聚焦测试，再跑回归测试。
5. 启动本地页面做最小交互验证。

---

## 9. 结论

这次采用“点击选中 + 长按拖动重排”的方案，重点不是继续修补旧拖动，而是把旧的交换式移动逻辑换成真正的插入式重排逻辑。

这样更贴近你的使用习惯，也更容易把之前一直没解决的移动问题一次性收住。
