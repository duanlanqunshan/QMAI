# QMAI GitHub 自动更新与发布设计文档

> 日期：2026-05-30
> 目标仓库：https://github.com/Mochocyang/QMAI
> 发布方式：GitHub Releases

---

## 1. 背景与目标

当前 `QMAI` 已经有 Windows 可执行文件打包流程，但还没有一套正式的：

1. 发布到 GitHub 的标准流程
2. 软件启动后自动检查新版本的能力
3. 面向外部用户的 README 发布说明

本次要解决的问题是：

1. 继续使用公开仓库 `Mochocyang/QMAI`
2. 使用免费的 GitHub Releases 作为安装包与更新元数据托管位置
3. 让本地 `QMaiWrite.exe` 能检测新版本并执行更新
4. 在 GitHub 仓库中补齐与当前软件状态相符的 README 文档

本次目标明确限定为“接入免费 GitHub 更新链路并补齐发布文档”，不扩展到付费 CDN、国内镜像或多平台自动发布。

---

## 2. 方案结论

采用以下固定方案：

1. 源码仓库继续使用 `Mochocyang/QMAI`
2. 发布渠道使用 GitHub Releases
3. 客户端更新能力使用 Tauri 2 的 updater 插件
4. 更新元数据使用静态 `latest.json`
5. 发布时同步维护 README，说明软件定位、功能现状、下载安装方式、更新方式与注意事项

这套方案的优点是：

1. 完全免费
2. 与当前 Tauri 2 技术栈兼容
3. 后续如果要切换到国内对象存储，只需要改更新地址，不需要重做整套逻辑

已接受的限制是：

1. 中国区下载速度可能受 GitHub 网络情况影响
2. 发布流程前期仍需要人工上传 Release 文件
3. 当前阶段优先解决“能稳定发布和更新”，不追求一键全自动发版

---

## 3. 用户可见效果

接入完成后，预期行为如下：

1. 用户安装现有版本的 `QMaiWrite.exe`
2. 软件启动时或手动触发时检查 GitHub 上是否有更高版本
3. 检查到新版本后，界面用中文提示用户存在新版本
4. 用户确认后下载并安装更新
5. 发布页面中用户可以通过 README 了解软件当前用途、主要能力、下载方式和更新说明

本次要求所有用户可见提示、更新说明、README 内容均使用中文。

---

## 4. 实现边界

### 4.1 预计修改文件

本次允许修改的核心文件：

1. `package.json`
2. `src-tauri/Cargo.toml`
3. `src-tauri/tauri.conf.json`
4. `src-tauri/src/lib.rs`
5. 前端中与应用启动、设置或提示相关的最小必要文件
6. `README.md`
7. 新增一份中文发布操作文档

### 4.2 不做的内容

本次明确不做：

1. 不重构现有项目结构
2. 不改动与自动更新无关的业务逻辑
3. 不引入付费云存储
4. 不增加 Linux、macOS 发布链路
5. 不删除已有功能函数
6. 不改造成复杂的 CI/CD 自动发布系统

---

## 5. 技术设计

### 5.1 Tauri 更新链路

当前仓库已经是 Tauri 2 项目，但尚未接入 updater 插件。本次设计为：

1. 在 Rust 侧接入 `tauri-plugin-updater`
2. 在 Tauri 配置中补充 updater endpoint
3. 使用 GitHub Releases 中的 `latest.json` 作为更新元数据入口
4. 使用 Tauri 官方签名机制校验更新包

更新源按公开 GitHub 仓库设计，默认指向：

`https://github.com/Mochocyang/QMAI/releases/latest/download/latest.json`

### 5.2 版本发布文件

每次发布新版本时，GitHub Release 中需要至少包含：

1. Windows 安装包或可更新包
2. 对应签名文件
3. `latest.json`
4. Release 说明

如果当前便携版 `QMaiWrite.exe` 不是 Tauri updater 可直接消费的更新包，则需要在保持现有便携打包不丢失的前提下，额外输出 updater 所需发布产物。

### 5.3 前端更新交互

前端只做最小必要交互：

1. 启动后检查更新
2. 有更新时显示中文确认提示
3. 下载和安装阶段显示中文状态
4. 无更新或检查失败时不打断主流程

实现风格遵循“外科手术式修改”，优先复用现有弹窗、设置或状态提示能力，不新增复杂页面。

### 5.4 README 内容边界

README 必须和当前软件状态一致，不能写成未来规划文案。建议至少包括：

1. 软件简介
2. 当前主要功能
3. 当前技术栈
4. 本地开发与打包方式
5. Windows 下载与安装说明
6. 自动更新说明
7. 已知限制或注意事项

README 重点是“真实、可运行、与当前版本一致”，不是营销文案。

---

## 6. 风险与控制

### 6.1 主要风险

1. 当前项目已有便携版打包脚本，接入 updater 后可能出现“能打包便携版但不能生成更新产物”的分叉
2. GitHub Release 资源命名、签名文件和 `latest.json` 之间如果不一致，会导致客户端检查更新失败
3. 自动更新提示如果放置不当，可能影响原有启动流程
4. README 如果脱离当前实际功能，后续每次发版都容易失真

### 6.2 控制策略

1. 先明确当前打包产物与 updater 产物的关系，再做最小改动
2. 所有新增提示文案使用中文
3. 先做本地构建验证，再打包验证
4. 打包完成后核对 `release-portable/QMaiWrite.exe` 时间与大小，防止误把旧包当新包
5. README 只写当前已经存在或本次确定要接入的内容

---

## 7. 验证标准

本次完成标准定义为：

1. 项目源码可以通过必要的类型检查与相关测试
2. Tauri 工程成功接入 updater 配置
3. 仓库中出现一份与当前软件状态一致的 `README.md`
4. 仓库中出现一份中文发布操作文档，说明如何生成并上传 GitHub Release
5. 成功重新打包，并确认 `E:\QMAI\release-portable\QMaiWrite.exe` 是新产物

如果当前条件不足以直接完成真实 GitHub 发布，则至少要把发布前所需信息、缺口和用户操作步骤列清楚，不能含糊结束。

---

## 8. 推荐实施顺序

1. 先梳理当前打包链路与 Tauri updater 所需配置
2. 最小化接入 updater 依赖与配置
3. 在前端接入最小更新检查入口
4. 补齐 `README.md`
5. 补齐中文发布操作文档
6. 跑构建与相关验证
7. 重新打包并检查 exe 产物

---

## 9. 结论

本次采用“公开 GitHub 仓库 + GitHub Releases + Tauri updater”的免费方案，并将 README 一并纳入交付范围。这样可以先把 QMAI 的 Windows 发布与自动更新能力正式搭起来，同时保持后续可演进空间。
