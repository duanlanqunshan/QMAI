## 五、核心功能目标

你需要在 `llm_wiki` 源码基础上实现以下能力。

### 1. Novel Mode 小说模式

新增小说模式开关：
novel\_mode = true / false

小说模式开启后：

- UI 文案小说化
- 知识库语义变成章节库
- 文件语义变成大纲
- Entity 语义变成人物 / 地点 / 物品 / 组织 / 伏笔 / 事件
- Review 语义变成 AI 审稿
- Graph 语义变成小说图谱
- Search 语义变成剧情搜索
- Check / Lint 语义变成连贯性检查

小说模式关闭后：

- 尽量保持原 `llm_wiki` 行为不变

### 2. 大纲管理

原 `File / Source` 能力尽量复用为“大纲”。

大纲包括：

- 总大纲
- 分卷大纲
- 章节细纲
- 人物设定
- 世界观设定
- 势力设定
- 时间线设定
- 伏笔计划
- 写作风格要求

当没有大纲时，用户可以让 LLM 生成大纲。

### 3. 章节库

原 `Knowledge Base / Wiki Page` 能力尽量复用为“小说章节 / 小说档案”。

章节包括：

- 章节正文
- 章节草稿
- 正式章节
- 章节快照
- 章节摘要
- 章节相关实体
- 章节图谱关系

### 4. Context Engine 上下文引擎

这是本项目核心。

每次用户要求：

- 写章节
- 续写
- 改写
- 润色
- 审稿
- 检查
- 生成细纲

系统都必须先生成“小说上下文包”。

上下文包需要自动提取：

- 总大纲
- 当前卷大纲
- 当前章节细纲
- 上一章正文或摘要
- 最近 N 章摘要
- 相关人物状态
- 相关地点
- 相关组织
- 相关物品
- 相关伏笔
- 当前时间线
- 角色认知状态
- Canon 正史规则
- 图谱相关节点
- 关键词搜索结果
- 向量检索结果
- 相关历史章节快照

然后合并、去重、排序、压缩，生成结构化上下文包。

上下文包格式类似：
\# 小说上下文包

```
## 当前任务
用户本次要做什么。

## 当前章节目标
本章必须完成什么。

## 必须遵守的大纲
总大纲、分卷大纲、章节细纲中的强约束。

## 最近剧情摘要
最近 3～10 章摘要。

## 上一章结尾
上一章最后状态。

## 当前人物状态
本章相关人物的状态。

## 角色认知状态
谁知道什么，谁不知道什么，读者知道什么。

## 当前伏笔状态
未回收、已推进、已回收伏笔。

## 时间线
当前时间点与历史关键事件。

## 相关地点 / 组织 / 物品
本章可能涉及的设定。

## 禁止违背
Canon 正史规则。

## 写作风格
节奏、文风、视角、人称、禁忌。
```

### 5. Chapter Ingest 章节摄取

这是第二个核心。

每章写完后，不允许直接污染记忆库。

必须区分：
Draft 草稿
Canon Chapter 正式章节
Archived Draft 废弃草稿

只有用户确认“保存为正式章节”后，才能触发 Chapter Ingest。

正式章节保存后，系统必须自动提取：

- 章节摘要
- 出场人物
- 出场地点
- 出场组织
- 出场物品
- 关键事件
- 人物状态变化
- 人物关系变化
- 角色认知变化
- 新增伏笔
- 推进伏笔
- 回收伏笔
- 新增设定
- 时间线事件
- 冲突变化
- 章节结尾钩子
- 图谱节点
- 图谱关系边
- 搜索索引文本
- 向量索引文本

并生成 `chapter_snapshot`。

### 6. 章节快照

每章保存后，必须生成结构化章节快照。

可以先使用 JSON + Markdown 双格式，也可以优先复用 `llm_wiki` 原存储结构。

示例字段：
{
"chapter\_id": "chapter-128",
"summary": "",
"characters": \[],
"locations": \[],
"organizations": \[],
"items": \[],
"events": \[],
"character\_state\_changes": \[],
"relationship\_changes": \[],
"knowledge\_changes": \[],
"foreshadowing\_changes": \[],
"new\_canon\_facts": \[],
"timeline\_events": \[],
"conflicts": \[],
"ending\_hook": "",
"graph\_nodes": \[],
"graph\_edges": \[]
}

### 7. 小说图谱

请复用 `llm_wiki` 原 Graph 能力，不要重写图谱系统。

小说模式下支持以下节点类型：
Character 人物
Location 地点
Organization 组织 / 势力
Item 物品
Event 事件
Chapter 章节
Outline 大纲
Foreshadowing 伏笔
Secret 秘密
Conflict 冲突
TimelinePoint 时间点
CanonRule 正史规则
Concept 概念

支持以下关系类型：
APPEARS\_IN 出场于
HAPPENS\_IN 发生于
BELONGS\_TO 属于
HAS\_ITEM 持有
ENEMY\_OF 敌对
ALLY\_OF 合作
SUSPECTS 怀疑
HIDES\_FROM 隐瞒
KNOWS 知道
DOES\_NOT\_KNOW 不知道
ADVANCES\_FORESHADOWING 推进伏笔
RESOLVES\_FORESHADOWING 回收伏笔
CREATES\_FORESHADOWING 新增伏笔
CAUSES 导致
REVEALS 揭示
AFFECTS 影响
LOCATED\_AT 位于

### 8. AI 审稿与连贯性检查

复用原 Review / Lint / Check 能力，小说化为：

- AI 审稿
- 连贯性检查
- 崩坏检查
- 伏笔检查
- 时间线检查
- 人设检查

检查维度包括：

- 是否违背总大纲
- 是否违背分卷大纲
- 是否违背章节目标
- 是否人设崩坏
- 是否人物动机不一致
- 是否时间线错误
- 是否地点错误
- 是否能力体系崩坏
- 是否伏笔遗忘
- 是否提前泄露秘密
- 是否角色知道了不该知道的信息
- 是否新增未登记设定
- 是否剧情水文
- 是否缺少章节钩子

检查结果应尽量结构化输出：
{
"severity": "warning",
"type": "character\_consistency",
"message": "主角在本章知道了此前未被揭示的信息",
"evidence": "正文片段",
"related\_memory": "第32章快照",
"suggestion": "改为主角怀疑，而不是确认"
}

### 9. 混合检索

不要只依赖 embedding。

长篇小说上下文检索必须使用：
向量搜索 + 关键词搜索 + 图谱搜索 + 最近章节窗口 + Canon 规则

Embedding 用于语义相似检索，但以下内容必须结构化保存：

- 人物状态
- 伏笔状态
- 时间线
- 角色认知
- Canon 正史规则
- 章节快照
